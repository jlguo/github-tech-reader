# OpenSource-Book-Generator 开源项目技术书籍生成 Skill
## 文档版本：V1.0
## 适配场景
读取任意开源项目源码/仓库，复刻《Claude Code 源码解析》橙皮书风格，产出**重工程决策、架构设计**的结构化技术书籍；支持 Plugin / Command / Multi-Agent 协同使用，全项目通用、可重复执行。

## 前置约束（全局强制规则）
### 1. 核心创作原则
1. 不逐行翻译、逐行解析代码，**聚焦技术选型、架构设计、工程决策、取舍逻辑、历史问题**；
2. 所有分析结论必须标注来源：源码路径、配置文件、注释、Commit/Issue、官方文档；
3. 全书面向**进阶开发者、架构师、二次开发人员**，拒绝入门科普、流水账式描述；
4. 固定分析版本：仅基于指定 Tag / Commit / 分支编写，明确标注版本边界，区分历史迭代。

### 2. 项目边界规则
1. 分析范围：仅主项目源码，自动过滤第三方依赖、测试用例、Demo 示例、外部插件；
2. 数据统计：代码行数、文件数量均排除依赖与测试代码；
3. 内容禁止：不涉及项目隐私数据、未公开服务、外部涉密接口。

### 3. 统一行文范式（全书所有章节强制使用）
所有模块/小节统一四段式结构，复刻原版橙皮书风格：
1. **模块定位**：该模块/功能的核心作用、业务职责；
2. **源码实现**：代码组织形式、核心文件、目录结构、关键组件；
3. **设计决策(Why)**：选型原因、痛点解决、方案对比、历史踩坑、安全/性能考量；
4. **复用价值**：可迁移的设计思路、通用避坑点、其他项目参考建议。

### 4. 统一排版规范
1. 数据类、对比类内容**强制使用表格**；
2. 源码路径、函数名、常量、配置项做加粗标注；
3. 每个章节末尾增加「本章核心收获」板块；
4. 区分三类内容：客观事实（源码原样）、设计分析（推理解读）、总结观点（经验提炼）；
5. 标题层级严格统一：一级标题/二级标题/三级标题，不自定义层级。

## 一、整体书籍架构（固定4大板块 + 12标准章节）
全书框架通用，适配**命令行工具、后端服务、前端框架、中间件、算法库**等所有开源项目；小型项目可合并子章节，不改动主板块。

> PART Ⅰ 项目全景（基础认知 · 第1~2章）

> PART Ⅲ 高级架构（深度特性 · 第9~11章）
> PART Ⅳ 工程方法论（总结提炼 · 第12章）

## 四、配套 Command 命令（人机交互入口）
封装快捷指令，拆分任务、支持断点续编，适配全流程：
1. `/book-generate-full`：一键全流程生成完整书籍（拉取源码 → 分析 → 排版输出）
2. `/book-init`：仅生成前言、数据看板、PART Ⅰ 全景章节
3. `/book-core`：仅生成 PART Ⅱ 核心模块章节
4. `/book-advance`：仅生成 PART Ⅲ 高级架构章节
5. `/book-summary`：仅生成 PART Ⅳ 方法论总结章节
6. `/book-check-style`：全文格式、风格、规范校验
7. `/book-export`：导出为 Markdown / 纯文本文件
8. `/book-clear-cache`：清空源码缓存，切换项目/版本

## 五、Multi-Agent 角色分工（协同规则）
### 5.1 角色定义（4类固定角色）
1. **Coordinator 总调度（Leader）**
   - 职责：接收命令、拆解任务、分配章节、全局进度把控、异常熔断、最终内容汇总
   - 权限：全局只读，统筹所有Agent，维护项目全局信息快照

2. **Source-Agent 源码解析员（Teammate）**
   - 职责：对接 Plugin 数据源，编写 PART Ⅰ、PART Ⅱ 基础&核心章节
   - 能力：目录解析、模块拆解、技术选型分析

3. **Arch-Agent 架构分析师（Teammate）**
   - 职责：专攻高阶内容，编写 PART Ⅲ 高级架构、版本、灰度相关章节
   - 能力：多实例分析、编译期设计、演进策略解读

4. **Review-Agent 审核校对员（独立角色）**
   - 职责：基于 Skill 规则校验格式、风格、逻辑；补充章节收获、修正错误
   - 权限：全文编辑，最终质量收口

### 5.2 协同工作流
用户触发 Command → Coordinator 拆解任务 → 分发至 Source-Agent / Arch-Agent 并行创作 → 内容汇总 → Review-Agent 审核 → Coordinator 整合全书 → 导出文件

### 5.3 冲突规避规则
1. 所有Agent仅读取 Plugin 输出的结构化数据，**不直接操作原始源码**，避免文件锁冲突；
2. Coordinator 统一分发全局术语、模块定义，杜绝章节上下文矛盾；
3. 长文档采用「分片上下文」传递，控制 Token 消耗。

## 六、Plugin 配套能力（底层数据源）
### 6.1 Plugin 核心功能
1. 源码拉取：支持 Git 仓库、本地目录，锁定指定 Tag / Commit / 分支
2. 批量扫描：统计文件数、代码行数、目录结构，自动生成数据看板
3. 静态解析：识别技术栈、入口文件、注释、配置、特性开关
4. 内容抽取：按模块拆分源码片段、关键常量、接口定义
5. 格式兼容：支持 TS / Go / Java / Python / JS 等主流编程语言

### 6.2 数据输出要求
输出**结构化JSON/文本素材**，过滤依赖与测试代码，附加文件路径标注。

## 七、风险规避规则（全局兜底）
1. 风格僵化：主框架固定，子章节支持合并/新增，适配极简项目与超大型项目；
2. 素材错误：Plugin 输出素材二次自检，异常自动上报 Coordinator；
3. Token 过载：启用 Micro 微压缩，分片传递上下文；
4. 并发冲突：全局文件锁，禁止多Agent同时读写原始文件；
5. 内容空洞：强制「设计决策」板块，无明确设计思路则标注「官方未公开设计考量」。
